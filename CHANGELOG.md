# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [1.0.0] - 2026-05-29

The first stable release. Folds the v0.6.0 work (bulk operations, labels, threads) together with an extensive correctness, security, and ergonomics hardening pass.

### Breaking changes

- **`read_message` body is now fenced with `[BEGIN UNTRUSTED EMAIL BODY]` / `[END UNTRUSTED EMAIL BODY]` markers.** Wire-format change: the email body section in the response is wrapped in explicit untrusted-content sentinels so downstream prompt templates can identify and isolate the attacker-controlled region via regex. Callers parsing the prior format will need to update — the body content itself is unchanged, but the surrounding `--- Body (...) ---` line is now followed by `[BEGIN UNTRUSTED EMAIL BODY]\n<body>\n[END UNTRUSTED EMAIL BODY]`. Headers, snippets, and other structured fields are unaffected.
- **`top_senders` now defaults `excludeSelf: true`.** The previous default included rows for the authenticated user, which produced misleading "who emails me" tables when scanning `All Mail` (since All Mail includes Sent). Callers who relied on self-inclusion should pass `excludeSelf: false` explicitly. The new response also carries a `direction: "self"|"received"` tag per row.
- **`sanitizeHtml` defaults to `true` on send / reply / reply-all / forward / save_draft.** Earlier builds defaulted to `false` to preserve full-fidelity HTML for trusted senders, but agent-driven workflows are by definition acting on user prompts and the safer-by-default posture is to scrub. Trusted-content callers must explicitly pass `sanitizeHtml: false`. Strips `<script>`, event handlers, inline `style`, disallowed tags, and remote `<img>` beacons; keeps text formatting, mailto/http/https links, and `cid:`/`data:` inline images.
- **`save_draft` no longer accepts a `folder` parameter.** Drafts always land in the `\Drafts` special-use folder (resolved at runtime; falls back to literal `Drafts`). Previously a malicious or confused caller could land a `\Draft`-flagged message in INBOX or any other path.
- **`rename_folder` is now restricted to `Folders/` and `Labels/` namespaces on both `from` and `to`.** Without this guard, `rename_folder(from: "INBOX", to: ...)` succeeds against Proton Mail Bridge — relocating INBOX's contents and triggering an auto-recreation that wipes UIDs. Path-traversal segments (`.` / `..`) are also rejected on both ends, matching `delete_folder`.
- **`create_folder` rejects `Labels/...` paths** with an actionable error pointing at the dedicated `create_label` tool. Previously `create_folder("Labels/X")` worked but muddied tool discovery. `create_folder` also now rejects `.`/`..` segments at create time (matching the pre-existing rejection at delete time).
- **`bulk_delete` with `permanent: true` now requires `confirm: true`.** A live permanent expunge is irreversible — there is no Trash to recover from — yet the prior API let one `match` hard-delete every match with only an optional `dryRun`. The expunge now requires an explicit `confirm: true` (mirroring `empty_folder`, which already gates comparable destructiveness). `dryRun: true` still previews without confirmation, and soft-delete-to-Trash (the default, `permanent` omitted/false) is unchanged. Callers passing `permanent: true` without `confirm: true` now get an actionable refusal instead of a deletion.

### Added (v1.0.0)

- **`list_messages.sortByUid`** — Opt-in flag to order by UID descending (arrival order, newest first) instead of date. The default date sort is paginated by a UID cursor (`beforeUid`), which is only approximate when UID order disagrees with date order — in `All Mail` or folders holding moved messages, date-mode page boundaries (and the internal 500-message scan cap, which takes the highest UIDs then date-sorts) can skip or reorder messages. `sortByUid: true` selects the highest `limit` UIDs *before* fetching, so each page is a contiguous UID window and `beforeUid` pagination is exact — no skips or duplicates. `INBOX` (append-only, UID ≈ date) is unaffected either way; the date-mode pagination hint now nudges toward `sortByUid` when more pages exist. For a precise date window, `search_messages` with `since`/`before` remains the tool.
- **`reply_all_email`** — Dedicated tool that replies to all original recipients (sender + TO + CC) minus self. Equivalent to `reply_email` with `replyAll: true`, exposed separately for agent discoverability. Self-exclusion now uses exact email-address match rather than a substring check.
- **`bulk_update_labels`** — Parity with `bulk_move`/`bulk_delete`/`bulk_update_flags`. Same XOR `uids`/`match` input shape and `dryRun` support. Adds are batched into one COPY per label; removes walk each label mailbox by Message-ID. Reports `{affected, notFound, notApplied}` so callers see when a remove was a no-op (label not applied to any UID) or an add was silently dropped.
- **`search_messages` attachment-name and -type filters** — New `attachmentName` (case-insensitive substring on filenames) and `attachmentType` (case-insensitive MIME-type prefix) on `search_messages` reuse the same 5 KB SIZE-floor + bodyStructure pipeline as `hasAttachment`, capped at 500 candidates. Closes the "find that PDF" use case without an extra round trip.
- **`forward_email.attachmentParts`** — Forward a subset of the original attachments by MIME part number. Discover part numbers via `list_attachments`, pass them as an array. Mutually exclusive with `includeAttachments: false`. Missing parts throw an actionable error listing the available set.
- **`download_attachment.saveTo`** — Optional relative path under `ALLOW_FILE_DOWNLOAD_DIR` (new env var). When set, the decoded bytes are written to that path inside the allowlist root and the response returns the file path + size instead of base64 — saves agents from blowing the token budget on multi-megabyte attachments. Path safety: rejects absolute paths, `..` traversal, and symlink escapes via realpath of the parent directory.
- **`send_email` / `reply_email` / `reply_all_email` / `forward_email` accept `markdownBody`** — Markdown source is rendered to HTML by [marked](https://github.com/markedjs/marked) and sent as `text/html` with the existing multipart/alternative plain-text fallback. Mutually exclusive with `body`/`isHtml`. Chosen for its small footprint and synchronous parse API.
- **`update_message_flags` reports `notApplied`** — When Proton Mail Bridge silently drops a user keyword (e.g. `"Important_Tag"`) on STORE, the response now distinguishes flags that took effect from those the server didn't actually apply. Same shape as `update_message_labels`. `bulk_update_flags` reports the same per-batch via the new `notApplied: string[]` field.
- **`get_thread` dedupes by Message-ID** — Under Proton's label model, a single physical message can appear in INBOX and All Mail with different UIDs; the previous response counted both as separate thread messages. Now collapses by Message-ID, prefers the non-All-Mail copy as the canonical row, and surfaces the alternate locations via `mailboxCopies` and `otherFolders` so the cross-namespace state stays visible.
- **`send_email` retries the sent-copy lookup and surfaces Reply-To rewriting** — Sent-folder UID lookup resolves the path via the `\Sent` special-use annotation (with fallback to literal `"Sent"`) and retries SEARCH across 6 attempts spanning ~17s (was 5s earlier — Sent copies were observed arriving at ~10s under load). When the sent copy is found, the delivered Reply-To header is compared against the requested value; if Proton rewrote it OR stripped it entirely (which Proton does when the rewritten value would equal From), the response includes a `server rewrote Reply-To` / `server stripped Reply-To` note. If the retry window exhausts without finding the Sent copy, the response notes that verification didn't complete.
- **HTML sanitization on `send_email` / `reply_email` / `reply_all_email` / `forward_email` / `save_draft`** — When the body is HTML (either via `isHtml: true` or `markdownBody`), runs the body through a conservative allowlist via [sanitize-html](https://github.com/apostrophecms/sanitize-html). Strips `<script>`, event handlers, inline `style` attributes, remote `<img>` beacons, and disallowed tags; keeps text-formatting tags, links (mailto/http/https), and inline `cid:`/`data:` images. **Defaults to `sanitizeHtml: true`** (see the breaking-changes section above) — pass `false` to preserve full-fidelity HTML for trusted-content workflows. The success response notes when sanitization ran. Single source of truth for the allowlist lives in `validation.ts` and is exercised directly by unit tests.
- **`ALLOW_FILE_DOWNLOAD_DIR` environment variable** — Gates the new `download_attachment.saveTo` capability. Unset → saveTo is rejected; set to an existing directory → saveTo paths resolve inside it. Safer-by-default than allowing arbitrary filesystem writes from an MCP server.
- **`empty_folder.dryRun`** — Preview the count that would be permanently deleted without touching any mail, for parity with `bulk_delete`/`bulk_move`. `dryRun: true` bypasses the `confirm: true` gate (a preview is non-destructive) and the folder restriction still applies. Lets an agent confirm the blast radius before an irreversible expunge.
- **`dryRun` on the send-family tools (`send_email` / `reply_email` / `reply_all_email` / `forward_email`).** The send tools were the only destructive class without a preview — yet they're the only ones with irreversible *external* side effects — a `reply_all_email` can fan out to external addresses embedded in the original message's recipients, so a preview before a live call matters. `dryRun: true` runs all validation and resolves the complete recipient set (To/CC/BCC, plus the reply-all fan-out / forwarded-attachment count) and returns it **without sending** — no SMTP, no Sent-copy verification, and for `forward_email` no attachment-byte download. The preview leads with an `[outbound:*]` token (`self-only`, `external:N`, `self-only-ok`, or `would-block`) and lists any external (non-self) recipients explicitly, so an agent can confirm exactly who would receive the mail before a live call. Does not consume the send rate-limit budget.
- **`RESTRICT_OUTBOUND_TO_SELF` environment variable.** Opt-in outbound guard for throwaway/QA accounts: when set, the send-family tools refuse a live send to any recipient other than the authenticated account (`PROTONMAIL_USERNAME`), with an actionable error listing the blocked external addresses. There is no per-call override — the env var is the gate, so a prompt-injected agent can't bypass it. Off by default (blocking external recipients is wrong for a general-purpose mail server) and independent of `READONLY` (which disables sends entirely; this one still allows self-sends). `dryRun` previews the would-block outcome via the `[outbound:would-block]` token without sending. Recipient resolution lives in `validation.ts` (`externalRecipientAddresses`) with unit tests.

### Fixed (v1.0.0)

- **`list_messages` / `search_messages` snippets are now flagged as untrusted content.** `read_message` fences its body in `[BEGIN/END UNTRUSTED EMAIL BODY]`, but the `includeSnippet` previews (~200 chars of raw sender-controlled body, appended inline after the `— ` separator on each row) carried no such marker — a prompt-injection vector for agents that triage by snippet. When snippets are present the output now appends a single untrusted-content banner instructing the caller to treat any text after `— ` as data, not instructions. (Single banner rather than per-row fences — proportionate to a 200-char preview, and `includeSnippet` is opt-in.)
- **`read_message` with `preferHtml: true` emits a `[html:active-content]` token when the raw HTML carries a live payload.** `preferHtml` returns the verbatim wire HTML (bypassing `sanitizeEmailHtml` so the agent sees the true content); the untrusted-body fence marks it as data but doesn't signal that the data contains `<script>`, inline event handlers, `javascript:` URLs, frames, or remote `<img>` beacons. A new `detectActiveHtml` check (in `validation.ts`, unit-tested) flags those cases so a careful agent can refuse to render. `cid:`/`data:` inline images are not flagged (they don't phone home). Advisory only — no stripping, no behavior change to the returned body.
- **`top_senders` explains an empty table caused by `excludeSelf`.** On a self-to-self account (or any folder of purely outbound mail), the `excludeSelf: true` default filtered every row and the response read `(no senders found)` — indistinguishable from "folder empty / tool failed." When the table is empty but envelopes were scanned, the response now says `excludeSelf=true filtered out all N scanned message(s) … pass excludeSelf=false to include your own outgoing address`; a genuinely empty scan says so explicitly.
- **`reply_all_email` error on a self-only recipient set now points at `reply_email`.** Replying-all to a message you sent only to yourself correctly errors (nobody but self remains), but the bare "Reply-all has no recipients other than yourself" read as an opaque failure. The message now names the likely cause and suggests `reply_email` to reply to the sender.
- **`get_thread` description clarifies it walks the reply chain, not forwards.** Forwards don't set `In-Reply-To`/`References` back to the original (a forward starts its own conversation), so a forwarded copy never appears in `get_thread` output. The description now states this explicitly so callers don't read a 1-result thread as a bug when forwards exist.

- **`top_senders` no longer lets a spoofed `From` display name poison a sender bucket's label.** Buckets are keyed by email address, but the label previously took the display name from the *last* message scanned — so a single inbound email with `From: "Security Team noreply@bank.com" <attacker@x>` relabeled every message from that address with the spoofed name. Display names are attacker-controlled, so the bucket now uses the **most frequently observed** name for the address (ties break deterministically); a lone spoofed message can't outvote the legitimate majority, and the address is always shown alongside. Regression test covers the majority-vs-spoof case.
- **`delete_message` from Trash returns an actionable error instead of an opaque failure.** Soft-delete moves a message to Trash; when the message was *already* in Trash that MOVE is a server no-op that surfaced as the generic "Failed to move … the server returned no confirmation." The soft-delete path now detects a Trash source and returns `… is already in Trash — soft-delete is a no-op here. To remove it permanently, call delete_message again with permanent: true.`
- **`delete_folder` now reports where a non-empty folder's contents went.** Deleting a folder on Proton is non-destructive (contents relocate to All Mail; deleting a label just drops the tag), but the bare "Folder X deleted." read as if the messages were destroyed. The response now appends `Its N message(s) were relocated to All Mail (not deleted).` for folders, or `The label was removed from N message(s); the messages themselves are unchanged.` for labels. Count is read via a `status()` probe before the delete (read-only, doesn't select the mailbox).
- **`search_messages` surfaces its approximation caveats in the response.** The subject/body staleness footer now fires whenever a `subject`/`body` filter is used (not only on zero results — non-empty result sets can also be incomplete on freshly-indexed mail), matching the bulk-op behavior. A new caveat is appended when `hasAttachment`/`attachmentName`/`attachmentType` is used, noting the ~5 KB size floor can miss messages with very small attachments.
- **`create_folder` / `create_label` translate Proton's opaque "Command failed" into an actionable error.** `create_label("Archive")` fails with a bare `Error: Command failed` while `Important`/`Test` succeed — Proton enforces unique names across folders and labels, so a label can't reuse the system "Archive" folder name, but the bridge reports the conflict only as "Command failed". On a namespaced create that fails (and isn't an idempotent already-exists), we now consult the live `client.list()`: (1) if the exact path is already present, return idempotent success — a safety net for the case where the `status()` probe is unreliable on Proton label-backed mailboxes; (2) if a *different* mailbox already owns the same leaf name, report the collision explicitly (`a mailbox named "Archive" already exists … Proton won't let a label reuse a name held by another mailbox`); (3) otherwise wrap the bridge error in a generic-but-actionable message naming the likely causes. The collision call-out is evidence-based (it only fires when the list actually shows the conflict) — no hardcoded "reserved names" guesswork. Regression tests cover all three branches. (`create_label` is also idempotent — it behaves as a `create_or_get_label`.)
- **`list_folders` no longer tags selectable Proton labels as "(namespace — not a mailbox)".** Two bridge quirks made the old detection mis-fire. First, it treated `listed === false` as non-selectable, but `listed` means "appeared in LSUB, not LIST" — a subscription-state signal unrelated to selectability; Proton surfaces labels via LSUB with `listed: false` while they remain fully selectable. That branch is now gated on an absent status object so it can only catch a genuine container. Second, Proton Mail Bridge has been observed returning a populated label with BOTH a `\Noselect`/`\NonExistent` flag AND inline STATUS counts (via LIST-STATUS), so the flag alone is not authoritative — a populated label like `Labels/Receipts` was tagged "(namespace — not a mailbox) — 3 messages, 1 unread", contradicting its own count. Detection now treats positive evidence of selectability as authoritative over the flag: a mailbox reporting `messages > 0` (you cannot hold messages in something you can't open), or — even at zero messages — a childless (`\HasNoChildren`) mailbox that returned a STATUS object (a STATUS succeeds only on an openable mailbox; a genuine container carries `\HasChildren`), is never tagged a namespace container. Regression tests cover both flag variants, the empty-leaf case, and the genuine `\Noselect`/`\NonExistent` containers.
- **`get_thread` (Message-ID mode) reroutes All Mail-only members to their real storage folder + UID.** The default walk (INBOX + Sent + All Mail) surfaces a thread stored in a user folder (e.g. `Folders/Development`) only through its All Mail virtual copy, so the row paired a folder with an All Mail UID that is invalid in any real folder. An agent copying that UID into a single-folder tool (`move_message`/`delete_message`) would target the wrong message or no-op. `get_thread` now reuses the same `rerouteAllMailOrphans` step the thread-mutation tools use, so cross-folder threads report a folder+UID that actually pair. The reroute opens a short-lived connection only when orphans are present (no-op otherwise). Regression test asserts the delegation; the reroute's own rewrite behavior is covered by the thread-op tests.
- **Read tools reject non-selectable namespace containers instead of reporting them empty.** Proton Mail Bridge lets `SELECT` succeed on its top-level `Folders` / `Labels` namespace nodes and returns them empty, so `list_messages` / `search_messages` / `count_messages` / `folder_stats` presented them as ordinary empty mailboxes — undercutting the `(namespace — not a mailbox)` annotation `list_folders` already applies. On the zero-result path (only), these tools now consult `LIST` and, on detecting a `\Noselect` container, throw an actionable error (`"Folders" is a namespace container … pick a child mailbox`). Zero cost on the common path; a genuinely empty real folder still returns empty. Regression tests cover the container rejection and the empty-real-folder pass-through.
- **Attachment size is now labelled approximate, and an earlier "floored, matches exactly" claim is corrected.** The decoded-size estimate is inherently ±~2 bytes (the true count needs the decoded bytes; line wrapping and final-line padding shift it either way — observed +1 over and −2 under on real attachments), so it is NOT a strict floor as the earlier note claimed. `list_attachments` / `read_message` now display it with a leading `~` (e.g. `~43808 bytes`), and the `AttachmentMeta.size` / `decodedSizeEstimate` docs spell out that it's an estimate, not byte-exact. The exact saved size is still returned by `download_attachment` (`bytes.length` of the decoded payload). No behavior change to the inline cap.
- **`get_thread` now emits the `@FolderName` row tag in `uid` mode too**, matching `messageId` mode for output-format parity. uid mode is single-folder, so rows are tagged with the scoped `folder` (the `??` fallback never clobbers the per-row folders messageId mode sets from its cross-folder walk).
- **Thread dry-run previews now show the destination-folder UID for All-Mail-rerouted members, not the stale All Mail UID.** The `acrossFolders` reroute (`rerouteAllMailOrphans`) rewrote an orphan's `folder` to its real storage location (e.g. `Folders/Archive`) but left its `uid` as the All Mail UID — so `move_thread`/`delete_thread`/`flag_thread` with `dryRun: true` previewed pairs like `Folders/Archive: UID 189` where 189 was invalid in Archive (the real UID was 13). Live execution was always correct (it re-resolves UIDs by Message-ID per folder and never reads the previewed UID), but an agent copying the previewed UID into a single-message tool (`move_message`/`delete_message`) would target the wrong message or no-op. The reroute now captures the UID from the per-folder Message-ID SEARCH it already performs and rewrites `uid` alongside `folder`. Regression tests assert the UID is rewritten on both the `acrossFolders: true` and `acrossFolders: false` paths.
- **`[sent-copy:unverified]` now spells out "do not resend".** All four send tools share one identical verification path; replies/forwards sent later in a session miss the ~30s Sent-folder lookup window more often purely from cumulative Proton indexing lag (not a code difference). The token was being misread as a delivery failure, risking duplicate sends on retry. The unverified response now states explicitly that SMTP accepted the message and the token only reflects the Sent-copy lookup, with a "do NOT resend" instruction.
- **Reply-all to your own sent message no longer loops back to you.** `reply_all_email` (and `reply_email` with `replyAll: true`) excluded self from the CC list but not from the primary reply target (the original `From`) — so replying-all to a message you sent put your own address in `To`. Self-exclusion now applies to the entire recipient set: participants are gathered in priority order (sender → original To → original Cc → extra Cc), self is dropped from every slot, and the first survivor becomes the primary `To`. In the common case the sender stays primary (unchanged behavior); when the sender is you, the first original recipient is promoted so the reply reaches the people you wrote to. Errors cleanly if nobody but self remains. The reply-all recipient logic moved to `validation.ts` (`buildReplyAllRecipients`) with regression tests for both the self-loop fix and the unchanged normal case.
- **Attachment sizes are now reported as the decoded file size, not the base64-encoded length.** `list_attachments` / `read_message` reported the IMAP-encoded octet count, which is ~37% larger than the real file for base64 parts (e.g. a 42,434-byte README shown as 58,068). `size` is now the decoded estimate (encoded × 76/78 × 3/4, floored — **approximate, ±~2 bytes, not byte-exact**, since the true count needs the decoded bytes; it can land both slightly over and slightly under real files); a new `encodedSize` field carries the wire size. The inline-download cap compares `encodedSize` (the base64 payload is what actually costs tokens), and the over-cap error now shows both numbers so the displayed size and the threshold no longer contradict each other. Non-base64 transfer encodings pass through unchanged.
- **`download_attachment` gives an actionable error when `ALLOW_FILE_DOWNLOAD_DIR` doesn't exist.** A misconfigured/uncreated download dir previously surfaced a raw `ENOENT … realpath` from Node. `resolveAllowlistedPath` now resolves the allowlist root first and translates `ENOENT` into "directory does not exist … create it first," with a separate message for a missing `saveTo` subdirectory. Tested with a mock `realpath` that throws `ENOENT`.
- **README: corrected the `sanitizeHtml` default for `reply_email` / `reply_all_email`.** Both parameter tables wrongly listed the default as `false`; the actual schema default (and the send-section banner, and the breaking-change note) is `true`. An agent trusting those rows would have assumed reply bodies went out unsanitized. The other three tables (send/forward/save_draft) were already correct.
- **`forward_email` no longer marks the original message `\Answered`.** The forward handler sent `In-Reply-To: <original>` and `References: <original>` — `In-Reply-To` declares "this message answers X," so Proton's server set `\Answered` on the source, mislabeling un-replied mail in "needs reply" filters. A forward goes to new recipients outside the original thread, so both threading headers are now dropped and the forward starts its own conversation. (The server-side `\Answered` behavior is only confirmable against a live Bridge; the code-level header removal is pinned by the forward regression guard. Replies still set `In-Reply-To` correctly.)
- **`list_folders` counts now carry a staleness caveat in the output, not just the description.** The per-folder message/unread counts come from a cached IMAP STATUS that Proton Bridge serves stale — e.g. `INBOX: 2/0` when the live state (per `folder_stats`, `count_messages`, and `list_messages`) was `3/1`. A footer now travels with the result pointing callers at `count_messages`/`folder_stats` (which SELECT+SEARCH the live mailbox) for authoritative counts, since an agent acts on the number rather than re-reading the tool description.
- **Connection errors now surface an actionable category instead of an opaque `connection failed`.** When the Bridge was down, every tool returned the identical string `Error: connection failed` with no host, port, code, or cause. `sanitizeErrorMessage` now distinguishes `ECONNREFUSED` (→ "connection refused … is Proton Mail Bridge running?") from `ENOTFOUND` (→ "host not found … check IMAP_HOST/PROTONMAIL_HOST") and surfaces the structured `address:port` (not sensitive — no credentials). Credential redaction is unaffected. Tests updated.
- **Bulk ops warn when resolving by subject/body `match`.** All four `match`-taking bulk tools — `bulk_move`, `bulk_delete`, `bulk_update_flags`, `bulk_update_labels` — now append a staleness warning to their output (dry-run and live) when the target set is resolved from a `subject`/`body` match. Proton Bridge's content SEARCH lags fresh mail ~30–60s, so a "clean up / relabel everything matching X" intent can silently under-act on recent messages — the dangerous silent-zero case where `bulk_update_labels match:{subject:...}` returns 0 against mail that plainly exists. The warning steers callers toward `from:`/date filters or explicit `uids`. Applied to every bulk tool (not just the destructive ones) because the misleading zero hurts regardless of reversibility — `bulk_move` is reversible and still warns. Same lag the `search_messages` zero-result footer already covers, now surfaced on the bulk surface too.
- **`search_messages.hasAttachment` size floor lowered from 25 KB → 5 KB.** Proton stores small attachments (text/JSON/tiny images) under the prior threshold, so `search_messages(folder="Sent", hasAttachment=true)` could silently return zero results when matches existed. The 500-candidate cap downstream still bounds worst-case bodyStructure fetches. Tests updated to assert the new floor.
- **`download_attachment` size-cap error now includes a concrete `claude_desktop_config.json` example.** The prior message named `ALLOW_FILE_DOWNLOAD_DIR` in the abstract; the new wording shows the full `env` block + a sample `saveTo` invocation so callers can act on it immediately instead of opening the README. README's pre-flight callout now embeds the same example.
- **`list_folders` tags IMAP `\Noselect` namespace containers** (e.g. Proton's top-level "Folders" / "Labels" nodes) with `(namespace — not a mailbox)` instead of letting them appear as 0-message folders. These are easily mistaken for empty mailboxes one could plant messages into; the explicit annotation closes the footgun. New `noSelect` field on `FolderInfo`. Regression test asserts the tag for both `\Noselect` and `\NonExistent`-flagged entries.
- **`update_message_flags` / `bulk_update_flags` emit `[flags:partial]` token when flags were silently dropped.** Proton Mail Bridge has been observed to silently drop user keywords; the response always carried the `no-op (not applied)` clause, but it was easy to miss in prose. Same prefix pattern as `[sent-copy:*]` / `[fromName:audit]` so callers grepping for partial-success indicators get a uniform signal.
- **`send_email` / `reply_email` / `reply_all_email` / `forward_email` emit `[sanitize:off]` token when an HTML body was sent with `sanitizeHtml: false`.** The opt-out is documented and intentional, but in agentic contexts a downstream-compromised agent could set the flag to deliver attacker-controlled `<script>` tags. The response token creates a paper trail without changing behavior, matching the `[fromName:audit]` precedent.
- **Thread operations reroute All Mail-only members to their real storage folder.** Under Proton's label model All Mail is a virtual aggregate, not a real folder — so when a thread member lives in a user folder (Archive, Folders/X, etc.) outside the INBOX + Sent walk, the dedupe surfaces only the All Mail copy, and a `move_thread` / `delete_thread` / `flag_thread` mutation against All Mail silently no-ops (`0 affected, 1 notFound`). The `rerouteAllMailOrphans` helper (run in `previewThread`) detects rows with no non-All-Mail alternative, scans the user mailbox list for the Message-ID, and rewrites the row's folder to the actual storage location before the thread op groups by folder — for **both** `acrossFolders: true` and `acrossFolders: false` (the default resolves a user-folder seed through All Mail and would otherwise hit the same no-op). Pays the extra IMAP round-trips only when the standard walk missed. Dry-run output lists the real storage folder (e.g. `Folders/Archive: 7 UIDs`) instead of a misleading `All Mail`. Regression tests cover both modes and the no-op short-circuit when an alternative folder is already known.

### Added (v0.6.0 — folded into v1.0.0)

- **`bulk_move` / `bulk_delete` / `bulk_update_flags`** — Mutate many messages in one IMAP command. Each accepts either an explicit `uids` array (max 1000) or `match` criteria (XOR), supports `dryRun` for safe preview, and reports `notFound[]` for UIDs the SEARCH pre-check didn't see. `bulk_delete` defaults to soft-delete via the resolved Trash path.
- **`create_folder` / `rename_folder` / `delete_folder`** — Folder management. `create_folder` is idempotent on already-existing paths and emits an actionable error pointing at Proton's `Folders/` namespace when a root-level path is rejected. `delete_folder` is restricted to the `Folders/` and `Labels/` namespaces so system mailboxes (INBOX, Sent, Trash) can't be removed; non-destructive on Proton (folder contents relocate to All Mail, label removal leaves messages untouched).
- **`create_label` / `update_message_labels`** — Proton labels. `create_label` takes a bare name (e.g. `"Important"`) and prepends `Labels/` internally — surfaced as a dedicated tool for agent discoverability rather than relying on agents to infer the namespace from `create_folder`'s description. `update_message_labels` adds/removes labels on a message: add via IMAP `COPY` (additive — message stays in source folder), remove by `Message-ID` lookup + `messageDelete` inside the label mailbox. Adds are strict (missing label → `Label not found`); removes are idempotent no-ops when the label doesn't apply or doesn't exist.
- **`empty_folder`** — Permanently empties a folder via atomic UID `EXPUNGE` (`client.messageDelete`). Requires `confirm: true` and (by default) restricts target to Trash/Junk unless `allowAnyFolder: true`. **Not registered unless the new `ALLOW_EMPTY_FOLDER=true` env var is set.** Enabling this is **not recommended** — Trash is the user's last line of defense against destructive agent actions, and the convenience of skipping a 2-click cleanup in the web UI does not justify giving an LLM the ability to bypass it. Keep this disabled unless you have a specific automation that needs it.
- **`count_messages`** — Returns just a number, backed by IMAP SEARCH; no envelope fetch. Rejects `hasAttachment` at the schema layer (would require an envelope scan that defeats the count's speed promise).
- **`folder_stats`** — Total/unread plus scanned-envelope aggregations (oldest/newest/total bytes). Default `scanLimit: 5000`, max 20000. Response includes `scanned`/`truncated`.
- **`top_senders`** — Sender frequency table over a configurable date range. Collapses display-name variants of the same address. Default `limit: 20`, `scanLimit: 5000`.
- **`move_thread` / `delete_thread` / `flag_thread`** — Thread-level mutations composed from `getThreadByMessageId` + bulk\_\*. Default per-folder; `acrossFolders: true` walks INBOX+Sent+All Mail. Dry-run previews scope to the same folder set as the live operation.
- **`ALLOW_EMPTY_FOLDER` environment variable** — opt-in gate (default `false`) controlling whether `empty_folder` is registered. **Not recommended** — see the `empty_folder` note above. Independent of `READONLY`, which still disables every mutating tool including this one.

### Fixed (v0.6.0 — folded into v1.0.0)

- **`empty_folder` no longer leaves messages in `\Deleted` limbo.** The previous implementation cast around the type system to call a non-existent `client.expunge()` method, so every invocation flagged all messages `\Deleted` and then crashed before expunging. Now routes through `client.messageDelete`, which performs STORE + EXPUNGE atomically. After upgrading, a single `empty_folder` call sweeps any leftover limbo state (assuming you opt in via `ALLOW_EMPTY_FOLDER`).
- **Thread dry-run scope now honors `acrossFolders`.** `move_thread` / `delete_thread` / `flag_thread` previews previously always walked INBOX + Sent + All Mail regardless of the flag, which could mislead users about what the live operation would touch. Added `ImapService.previewThread` shared between the dry-run preview and live execution.
- **`count_messages` rejects `hasAttachment` instead of silently dropping it.** The Zod schema now omits the field, surfacing an `Unrecognized key` validation error so callers don't get a misleadingly large count. `search_messages` remains the supported path for attachment filtering.
- **`create_folder` returns an actionable error on Proton.** A root-level path (e.g. `"Receipts"`) previously surfaced the generic `Command failed`. Now translates to a hint pointing at the required `Folders/` namespace (e.g. `Folders/Receipts`).
- **`create_folder` is genuinely idempotent on Proton.** Re-creating an existing folder previously failed with a bare `Command failed` because Proton Mail Bridge doesn't set `ALREADYEXISTS` or include "already exists" in the message — `create_folder` now probes with `status()` after a failed create and reports `alreadyExists` when the mailbox is in fact already there.
- **`rename_folder` missing-source error is now translated on Proton.** Renaming a nonexistent source folder previously returned `Command failed` because Bridge doesn't annotate the error with `mailboxMissing`. A post-failure `status()` probe (mirroring the existing `move_message` pattern) now produces `Folder not found: <path>` while leaving genuine other failures untouched.
- **Bulk operation `notFound` accounting no longer lags reality.** `bulk_move`, `bulk_delete` (`permanent: true`), and `bulk_update_flags` previously identified phantom UIDs via IMAP `SEARCH`, which on Proton Mail Bridge can return stale results due to server-side indexing lag — leading to truthful deletions being reported as `notFound`. The pre-check is now a `FETCH` over the UID range, which only yields existing messages and is authoritative for per-UID state.
- **README parameter names match the live schema.** `create_folder` is documented as `path` (not `folder`); `rename_folder` is `from`/`to` (not `folder`/`newName`).
- **`update_message_labels` no longer reports false success on a missing label.** imapflow's `messageCopy` resolves `false` (rather than throws) when the destination mailbox doesn't exist, and Proton Mail Bridge returns a bare `Command failed` without the `mailboxMissing` signal — so the previous try/catch silently swallowed both cases. Each label-to-add is now pre-checked with `client.status()`; a missing label throws the documented `Label not found: <path>` error. Pre-check happens after the source-UID lookup, so a wrong UID still surfaces ahead of label errors.
- **`update_message_labels` distinguishes real removes from no-ops.** The return shape is now `{added, removed, notApplied}`. The tool response reports `removed: …` only for labels actually stripped from the message; labels that weren't applied (either because the message didn't carry them, or the label mailbox doesn't exist) are surfaced as `no-op (not applied): …` so callers aren't misled by optimistic wording.
- **Bulk-tool `dryRun` previews use the same FETCH pre-check as live runs.** Previously, when `uids[]` was passed explicitly, `bulk_move` / `bulk_delete` / `bulk_update_flags` dry-runs echoed the caller's list verbatim — inflating the impact count and omitting phantom UIDs that the live execution would correctly catch. Added `ImapService.filterExistingUids(folder, uids)` (public wrapper around the same `existingUidSet` helper) and wired it into all three dry-run branches. The `match`-based dry-run path is unchanged (it already runs a real `SEARCH`). Dry-run output now matches the live `notFound` accounting shape exactly.
- **Thread mutations no longer report false `notFound` on Proton's label cascade.** `move_thread` / `delete_thread` / `flag_thread` (with `acrossFolders: true`) previously snapshotted UIDs upfront via `previewThread`, then operated folder-by-folder. On Proton's label model, mutating one folder cascades into others (e.g., moving from INBOX to Trash strips the `All Mail` label too, since All Mail excludes Trash), so by the time we reached the next folder its snapshotted UIDs were stale — reported as `notFound` even though the underlying message had moved. Now `threadOp` gathers Message-IDs upfront and re-resolves the current UID set per folder at the moment we operate on it via the new `resolveUidsByMessageIds()` helper. Folders that have already cascaded report a clean `{ affected: 0, notFound: [] }` instead of phantom UIDs. `MessageSummary` gained an optional `messageId` field, populated by every fetch path that requests envelopes.

### Internal

- **`SpecialFolderResolver`** (`src/special-folders.ts`) — caches `\Trash`/`\Junk`/`\Archive`/`\Sent`/`\Drafts` paths from the first LIST. Concurrent first-callers share a single in-flight promise. Falls back to literal name when annotation is absent.

## [0.5.0] - 2026-04-20

### Added

- **`list_attachments` tool** — fetch attachment metadata (part numbers, filenames, types, sizes) without downloading the body. Composes with `download_attachment` for bulk extraction workflows.
- **`get_thread.messageId` input** — preferred over UID+folder; walks INBOX, Sent, and All Mail by default so cross-folder threads (the common case) are returned intact. New optional `folders` parameter lets callers override the default set. Output rows are tagged `UID X @FolderName` to disambiguate per-folder copies.
- **`read_message.showHeaders`** — when `true`, surfaces `In-Reply-To`, `References`, `Reply-To`, `List-Unsubscribe`, and `List-ID` under an `--- Extra Headers ---` section. Handles RFC 5322 folded continuation lines.
- **`read_message.stripUrls`** — when `true`, drops anchor URLs from stripped-HTML output so agents summarizing newsletters don't burn tokens on tracking URLs.
- **`save_draft.fromName` and `save_draft.replyTo`** — feature parity with `send_email`.
- **`forward_email.includeAttachments`** — defaults to `true` so forwards now carry original attachments as agents expect; set to `false` to forward the body alone.
- **`list_messages` pagination hint** — when the result count equals the requested limit, the response footer suggests the `beforeUid` value to use for the next page.

### Fixed

- **Silent false success on bogus UIDs** for `delete_message`, `update_message_flags`, and `move_message`. IMAP STORE/DELETE/MOVE succeed silently against missing UIDs; we now pre-check existence via `fetchOne` and throw `"Message UID N not found in <folder>"` (matching the pattern `read_message` already used). Applies equally to `list_attachments` and `download_attachment`.
- **Attachment content returned as body** in `read_message` for HTML messages that carried a `text/plain` attachment. The body-part selector previously matched on content-type only; it now skips parts with `Content-Disposition: attachment` so the body is the real body.
- **`forward_email` dropped original attachments.** Forwards now download each attachment from the source message and re-attach.
- **Folder-not-found errors are actionable across every folder-taking tool.** All `getMailboxLock` calls route through a shared helper that translates imapflow's canonical `err.mailboxMissing` / `err.serverResponseCode ∈ {NONEXISTENT, TRYCREATE}` signals into `"Folder not found: <path>"`. `move_message` destination failures now probe the destination with `client.status()` to distinguish missing folders from other failures. `saveDraft` append failures are translated the same way.
- **Flag whitelist in `validateImapFlag`.** Previously `\`-prefixed names were accepted wholesale, so IMAP would silently drop unknown system flags like `\Bogus`. Now we accept only RFC 3501 system flags (`\Seen`, `\Flagged`, `\Answered`, `\Draft`, `\Deleted`, `\Recent`) plus alphanumeric user keywords per RFC 3501. The error message lists the allowed set.
- **Date string validation** for `search_messages.since` / `search_messages.before` / `mark_all_read.olderThan`. New `isValidDateString` helper rejects anything that isn't a valid `YYYY-MM-DD` calendar date (catches `04-19-2026`, `2026-13-01`, `2026-02-29`, etc.).
- **`download_attachment` errors are now actionable.** A bogus part number throws `"Part X not found on UID Y in <folder>; known parts: [2, 3]"` (or `[(none)]` when the message has no attachments) instead of imapflow's opaque `"Command failed"`.
- **`mark_all_read` copy under `olderThan`.** When the filter matches zero messages, the response now says `"No unread messages older than <date> in <folder>"` instead of implying the folder has no unread mail.
- **BCC addresses are no longer echoed verbatim** in the `send_email` success response. Masked to `"and BCC to N recipient(s)"` so verbatim-logged output doesn't leak the BCC list.

### Changed

- Tool descriptions updated to document semantics that were previously implicit:
  - `update_message_flags` now lists both RFC 3501 system flags and user keywords.
  - `send_email.replyTo` and `save_draft.replyTo` note that Proton SMTP may rewrite non-authenticated values.
  - `search_messages.since` is documented as inclusive; `search_messages.before` and `mark_all_read.olderThan` as exclusive.
  - `read_message.maxBodyLength` surfaces its 100..500 000 range in prose.

## [0.4.1] - 2026-04-19

### Fixed

- **Critical: `list_messages` and `search_messages` no longer fail with "Connection not available"** on real IMAP connections. The v0.4.0 refactor introduced `return this.fetchSortAndLimit(...)` without `await`, which caused the outer `try/finally` to close the IMAP connection (via `client.logout()`) before the fetch iterator resolved. Unit tests passed because mocks have no real connection state. Fixed by adding `await` to both call sites and adding regression tests that simulate connection closure mid-fetch.

## [0.4.0] - 2026-04-17

### Added

- **`get_thread` tool** — find all messages in a conversation by walking In-Reply-To and References headers. Returns messages sorted chronologically (oldest first).
- **`mark_all_read` tool** — bulk mark all unread messages in a folder as read, with optional `olderThan` date filter. Single IMAP connection + single STORE command.
- **`save_draft` tool** — save an email as a draft without sending it. Uses IMAP APPEND to place a fully-formed message in the Drafts folder for user review.
- `send_email` now attempts a best-effort lookup of the sent copy UID in the Sent folder and includes it in the response when found.

### Changed

- **HTML-to-text conversion** now uses the `html-to-text` package instead of naive regex stripping. Preserves links as `text [url]`, formats lists with bullets, skips tracking pixels, and handles entities correctly.

### Fixed

- **Search and list ordering is now correct by date.** Replaced the 5x UID heuristic with a shared `fetchSortAndLimit()` helper that fetches all matching envelopes (up to 500) before sorting by date. This fixes the reported bug where `search_messages` and `list_messages` could return old messages instead of the most recent ones when UIDs didn't correlate with dates (e.g. moved messages).

## [0.3.0] - 2025-04-15

### Added

- `send_email`, `reply_email`, and `forward_email` now return the Message-ID in their response, allowing callers to correlate sent messages with what appears in the mailbox.
- `reply_email` includes quoted original message with attribution line (`On <date>, <sender> wrote:`). Opt out with `includeQuote: false`.
- `move_message` and `delete_message` (non-permanent) now return the new UID in the destination folder when the server supports the UIDPLUS extension.
- Truncation marker includes the original body length and a suggested `maxBodyLength` value to retrieve the full message.

### Fixed

- `list_messages` ordering is now consistent across back-to-back calls. Replaced racy sequence-number-based fetch with UID-based search, eliminating cases where recent messages could be missing or returned in inconsistent order.
